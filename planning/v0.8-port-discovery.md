# v0.8 — Port discovery via `~/.chorus/daemon.json`

**Goal:** decouple chorus's runtime ports from hardcoded literals so the daemon can shift to a free port when 7707/5050 are squatted (VSCode tunnel pre-forwards in WSL, lingering processes the user can't identify, sudo'd zombies). Today every consumer assumes `127.0.0.1:7707` for the daemon and `127.0.0.1:5050` for the cockpit; that coupling is what blocks us from auto-shifting safely.

Tracked in journal as **chorus-095** (architectural debt, recorded 2026-05-06).

## What's coupled today

| Consumer | Coupling site | What it assumes |
|---|---|---|
| MCP shim (stdio binary) | `src/mcp/client.ts:8` `DAEMON_BASE = "http://127.0.0.1:7707"` | Daemon on 7707 |
| Cockpit server (Next.js) | `src/app/api/daemon/[...path]/route.ts` | Daemon on 7707 |
| Cockpit browser | `<script>` injecting cockpit base URL | Cockpit on 5050 |
| `chorus stop` / `chorus status` | `src/cli/port-utils.ts` probes 7707 + 5050 | Both default ports |
| `chorus connect` / `chorus init` | Writes MCP config pointing at the chorus binary (NOT a port) | Already abstracted ✓ |

The last row is the win: user-facing MCP configs reference `chorus mcp` (stdio binary), not an HTTP URL. So **port discovery is a chorus-internal concern** — no user re-configuration after a shift.

## Design

### File: `~/.chorus/daemon.json`

```json
{
  "schemaVersion": 1,
  "daemonPort": 7707,
  "cockpitPort": 5050,
  "daemonPid": 12345,
  "cockpitPid": 12346,
  "startedAt": "2026-05-06T11:23:00Z",
  "version": "0.8.0"
}
```

Written by `chorus start` after both ports are bound. Atomic write: `tmp → rename`. Read by every other consumer at runtime.

### Port-pick algorithm (`chorus start`)

1. **Daemon: try preferred port first** (default `7707`, override via `CHORUS_DAEMON_PORT`).
   - If free → bind it.
   - If taken → probe `/api/v1/health` with **5s timeout** (WSL loopback is slow). If a healthy chorus daemon answers, exit 0 with the cockpit URL (the existing `alreadyRunningOnDefaultPort` path, just with the longer timeout).
   - If health probe fails or returns non-chorus → walk `7708..7720` looking for a free port. First free wins.
   - If 7707..7720 are all squatted → hard fail with the existing diagnostic ("here's how to find the squatter").
2. **Cockpit: same logic** for `5050..5063`.
3. **Write `daemon.json`** atomically once both processes have bound their ports.
4. **Print the chosen URLs** so the user knows where the cockpit lives even if it's not 5050.

### Read flow (consumers)

A new shared helper `src/lib/daemon-discovery.ts`:

```typescript
interface DaemonInfo {
  daemonPort: number;
  cockpitPort: number;
  daemonPid: number;
  cockpitPid: number;
}

export async function readDaemonInfo(): Promise<DaemonInfo>;
export function daemonBaseUrl(): Promise<string>;  // "http://127.0.0.1:<port>"
export function cockpitBaseUrl(): Promise<string>;
```

Resolution order:
1. Read `~/.chorus/daemon.json` → parse → liveness-check via `/api/v1/health` on `daemonPort`.
2. If file missing or stale (parse fail / health fail / pid dead): **fall back to defaults 7707 / 5050**. This preserves back-compat for old `daemon.json`-less installs and means the tool keeps working if the file is corrupted.
3. Cache the result in memory for the lifetime of the process (no per-call disk reads).

### Changes per consumer

- **`src/mcp/client.ts`** — replace `DAEMON_BASE` constant with `daemonBaseUrl()` call inside the fetch wrapper. Each MCP tool call resolves once and caches.
- **`src/app/api/daemon/[...path]/route.ts`** — same: `daemonBaseUrl()` not a literal.
- **Cockpit base URL in browser** — server component reads `cockpitBaseUrl()` and injects via env or window global. Today the value is implicit (the page is served from the cockpit itself); this matters for the CLI's "open browser at X" step.
- **`src/cli/port-utils.ts`** — `chorus stop` / `chorus status` read `daemon.json` instead of probing fixed ports. Stop also kills by PID from the file (more reliable than port-based PID lookup).
- **`chorus init`** — keep emitting `chorus mcp` (stdio) in MCP configs; no URL change needed.

### Single-instance guarantee

`daemon.json` doubles as a lockfile. On `chorus start`:
- If file exists AND `/api/v1/health` returns chorus on the recorded port → "already running, see <cockpit URL>".
- If file exists AND health probe fails → file is stale, overwrite.
- If file missing → fresh start.

Race: two `chorus start` invocations launched simultaneously. Mitigation: atomic `tmp → rename` write + post-write read-back; loser detects the winner's file and exits with "already running" pointing at the winner. Acceptable because the loser's daemon process is killed before the file is committed (start sequence: pick port → bind → write file; if write loses, kill local daemon).

### Backward compat

- Old MCP configs (created by pre-0.8 `chorus init` / `chorus connect`) launch `chorus mcp`, which now reads `daemon.json`. **No user action required after upgrade.**
- A v0.7.x daemon left running when v0.8 is installed: `daemon.json` doesn't exist (v0.7 didn't write it). v0.8 `chorus start` probes 7707, finds healthy chorus, exits 0 — same as today. Once user restarts cleanly under v0.8, file gets written.
- `CHORUS_DAEMON_PORT` env override still works — sets the **preferred** starting port for the walk, not a hard requirement.

## Open decisions (need confirmation)

1. **Cockpit port shift: yes or only daemon?** Daemon shift solves the WSL VSCode case for MCP. Cockpit shift solves the same case for the browser-facing UI. Recommend yes — same algorithm, ~10 extra lines, removes a parallel failure mode. *(Default: yes)*

2. **Hard-fail port range — 7707..7720 (14 ports) and 5050..5063 (14 ports), or wider?** Wider = more resilient but more confusing if user has unrelated software at those numbers. Recommend 14. *(Default: 14 each.)*

3. **What does `chorus mcp` do if daemon.json is missing AND port 7707 has nothing healthy?** **Decision: auto-start the daemon.** The MCP shim is invoked by editor (Claude Code / Codex / Cursor), so a connection-refused error is bad UX — the user expects "/chorus" to just work. Auto-start is the same pattern Docker / Spotify use.

   **Spec:**
   - On the first tool call, if `daemonBaseUrl()` resolution finds no live daemon, the shim spawns `chorus start` (daemon only, no cockpit) as a detached background process. Use `process.execPath` to locate the chorus binary so the spawn doesn't depend on PATH that the editor passed through.
   - Poll for `daemon.json` to appear AND `/api/v1/health` to pass, **10s timeout** (WSL slow). Stderr line: `chorus: daemon not running, auto-starting…` (stdout is reserved for the MCP protocol).
   - Race between parallel MCP calls during cold start: first to spawn wins via the existing `chorus start`'s "already running" detection (chorus-094); the losers see the file appear and proceed normally.
   - On auto-start failure (port walk exhausted, binary missing, etc): surface the error to the editor as the MCP tool's error response — `chorus: failed to auto-start daemon: <reason>`. Don't retry-loop; the user reads it once and acts.
   - Cockpit: NOT started automatically. User invoked via MCP, not via UI. They can run `chorus start --ui` separately if they want the cockpit.
   - Override: `CHORUS_AUTOSTART=0` env var disables this entirely (back to "Run `chorus start` first" error). For users who want explicit control.

4. **Liveness check — health probe alone, or PID + health?** PID alone is unreliable (PID reuse). Health alone is fine but slow on miss. Recommend health probe with 1s timeout (this is the local re-resolution case, not the start-up case where we want 5s). PID check is a cheap pre-filter. *(Default: PID-alive pre-check + 1s health probe.)*

## Non-goals for v0.8

- Multi-daemon-per-host (each user under different `CHORUS_HOME`). Already works today via env override; design preserves that.
- IPv6 / non-loopback bindings. Daemon stays on 127.0.0.1.
- Graceful in-place port migration of a running daemon (would need cockpit re-discovery mid-session). Not worth it.
